sftp connection closed

Author: Chloe Date: June 8, 2020  1:33:43

1、现象：

工作中出现应用反馈两台服务器192.12.3.27、192.12.3.28，其中一台服务器无法使用sftp，另外一台可以使用，提示connection closed

2、什么是sftp：

  • sftp（英语：SSH File Transfer Protocol，也称 Secret File Transfer Protocol，中文：安全文件传送协议，英文：Secure FTP 或字母缩写：SFTP），提供文件访问、传输和管理功能的网络传输协议。由互联网工程任务组（IETF）设计，透过 SSH 2.0 的扩展提供安全文件传输能力，但也能够被其他协议使用。是安全文件传送协议，可以为传输文件提供一种安全的网络的加密方法。

  • sftp 与 ftp 有着几乎一样的语法和功能。

  • 即使 IETF 在网络草案数据阶段时，这个协议是在 SSH-2 文件中描述，它能够使用在许多不同的应用程序，例如安全文件传输在传输层安全（TLS）和传输信息管理于虚拟专用网应用程序。这个协议是假设运行在安全信道，例如 SSH，服务器已经认证客户端，并且客户端用户可利用协议。

  • 与早期的 SCP 协议只允许文件传输相比，SFTP 允许对远程文件进行更广泛的操作——这更像一个远程文件系统协议。一个 SFTP 客户端与 SCP 的能力比较包括额外的恢复中断传输，目录列表和远程文件移动。

  • SFTP是一种安全的文件传输协议，一种通过网络传输文件的安全方法；它确保使用私有和安全的数据流来安全地传输数据。

  • SFTP要求客户端用户必须由服务器进行身份验证，并且数据传输必须通过安全通道（SSH）进行，即不传输明文密码或文件数据。它允许对远程文件执行各种操作，有点像远程文件系统协议。SFTP允许从暂停传输，目录列表和远程文件删除等操作中恢复。

  • 和传统的ftp相比：

  1. FTP 基于TCP来传输文件，明文传输用户信息和数据。
  2. SFTP 基于SSH来加密传输文件，可靠性高，可断点续传。

3、解决思路：

  1. 是否存在密码过期，chage -l 用户名：发现密码是新置的，并且用ssh可以登陆，不存在密码问题。
  2. 是否是做了用户登陆限制，查看是否配置了sftp和ssh走了不同端口，发现/etc/ssh/sshd_config和/etc/ssh/ssh_config 文件中并没有任何port的设置，暂时没有现象表明走的不同的端口。
  3. 百度大法：搜索connection closed，

    但是发现实际配置已经是： 所以，显然不是这个原因造成了sftp无法连接。
  4. root能否连接sftp，结果是成功的，说明sftp正常。
  5. 是不是这个test用户有做什么限制，新建一个普通用户是否能够连接sftp，结果也是一样的，connection closed。
  6. 查看日志：/var/log/secure和/var/log/messages两个日志，查找sftp关键字，然后看到日志里有： 然后百度搜了一下，也没明白到底是什么。。。第一反应是中病毒了？？？
  7. 然后我突然灵光一闪，这个sftp走的的传输协议，如果sftp不正常，那scp是否正常？ 喔唷～看来可能是这个有问题，这个提示放到百度里，一般搜到的是：
    No such file or directory
    too short之类，观察报错信息，看着就是权限不够，结合前面测试的root能登陆，普通用户不能登录，可以推测某个库文件，权限只有750权限，造成root用户能够使用sftp，而普通用户无法sftp 到这几个系统目录下查看权限，果然对应文件的其他用户权限是0，于是我改了一下权限 测试通过！！！解决了！！！
    然后为了确保无疑，去正常的服务器上查看，别的服务器上该文件确实是755的权限。
    libcrypto.so.1.0.0这个文件看着像是加密解密的文件.
    后来再跟应用确认的时候，应用表示当时他们在部署mysql的时候，可能碰过这个文件，导致该文件的权限出现异常。
